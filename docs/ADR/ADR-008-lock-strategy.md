# ADR-008. 좌석 홀드 락 전략 — 비관적 락 vs 낙관적 락 vs DB 유니크 제약

## 상태

검토 중 (인간 확정 시 `확정`으로 변경)

## 컨텍스트

Phase C-1(좌석 홀드)의 핵심 요구는 **"같은 (event, seat)에 동시 요청이 몰려도 정확히 한 명만 홀드에 성공"** 하는 것이다.
이 프로젝트의 핵심 주제가 동시성 제어이므로, 이 결정이 가장 중요한 ADR이다.

전제로 ADR-001에서 홀드를 독립 테이블로 분리하지 않고 Seat 행에 `status`, `held_by`, `held_until`로 흡수하기로 했다.
따라서 경합 단위는 **단일 Seat 행(seat.id)** 이다.

master_plan이 제시한 세 가지 후보 락 전략을 이 프로젝트의 실제 맥락(단일 행 흡수 모델, 인기 좌석 집중 시나리오)에 맞춰 비교한다.

현재 코드/스키마 기준:
- `Seat` 엔티티: `status`(AVAILABLE/HELD/CONFIRMED), `heldBy`, `heldUntil` 보유. `@Version` 없음, 도메인 메서드 없음(불변).
- `seats` 테이블: `uq_seats_event_label (event_id, label)` 유니크, `idx_seats_status_held_until` 인덱스.
- `SeatRepository`: `@Lock` 미적용. 만료 스캔용 `findByStatusAndHeldUntilBefore` 존재.

## 근거 (비교 분석)

| 전략 | 구현 방식(본 프로젝트) | 경합 시 동작 | 장점 | 단점/리스크 |
|---|---|---|---|---|
| **비관적 락** | `SeatRepository`에 `@Lock(PESSIMISTIC_WRITE)` 추가 → `SELECT ... FOR UPDATE`로 seat 행 잠근 뒤 status 검사·전이 | 후행 트랜잭션이 락 해제까지 **블로킹(대기)** 후 갱신된 status 확인 | 단일 행 흡수 모델(ADR-001)과 직결, 로직 단순·직관, 애플리케이션 재시도 로직 불필요 | 락 보유 동안 DB 커넥션 점유, 경합 폭주 시 대기열·타임아웃 발생, 락 순서 어긋나면 데드락 |
| **낙관적 락** | `Seat`에 `@Version` 컬럼 추가, status 전이 시 버전 비교. 충돌 시 `OptimisticLockException` 발생 | 후행 트랜잭션 **즉시 실패** → 애플리케이션이 재조회·재시도 | 락 미보유로 저경합 시 처리량 유리, 커넥션을 빨리 반환 | **고경합(인기 좌석 1개에 N명)에서 충돌·재시도 폭증 → 라이브락 위험**, `@Version` 컬럼 추가(스키마 변경), 재시도/실패 처리 로직 부담 |
| **DB 유니크 제약** | 홀드를 별도 테이블 `holds(event_id, seat_id) UNIQUE`에 INSERT, 중복 키 위반으로 동시 홀드 차단 | 후행 INSERT가 **`DuplicateKeyException` 즉시 실패** | DB 원자성에 위임, 명시적 락 코드 불필요 | **ADR-001(행 흡수 결정)과 충돌** — 별도 홀드 테이블 신설 필요, TTL 만료·해제 로직이 seats와 holds로 분산, 스키마/일관성 비용 큼 |

추가 분석 포인트:
- **시나리오 특성:** 본 서비스는 "인기 좌석 1개에 다수가 동시 집중"되는 고경합·단일 행 패턴이 핵심이다. 이 패턴에서 낙관적 락은 대부분의 요청이 충돌·재시도로 귀결되어 라이브락·CPU 낭비 위험이 가장 크게 드러난다.
- **ADR-001 정합성:** DB 유니크 제약 방식은 홀드를 별도 테이블로 다시 분리해야 하므로 ADR-001(행 흡수) 결정을 사실상 되돌리게 된다. 일관성 유지 비용이 크다.
- **C-2 확정 연계:** 비관적 락은 홀드 시점에 잡은 행 락이 확정(C-2) 트랜잭션의 좌석 상태 전이까지 락 경계를 명확하게 이어준다.
- **측정 연계:** 실제 처리량/대기 수치는 정성 비교만으로 단정할 수 없다. Phase D 부하 테스트에서 비관적 락의 처리량 상한을 수치화한다(이 ADR은 정성 비교까지).

## 결정 (Claude 추천안 — 인간 확정 대기)

**추천: 비관적 락(`PESSIMISTIC_WRITE`)**

```java
// SeatRepository
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("select s from Seat s where s.id = :id")
Optional<Seat> findByIdForUpdate(@Param("id") Long id);
```

추천 근거 요약:
1. ADR-001의 단일 행 흡수 모델과 가장 자연스럽게 결합한다(`SELECT ... FOR UPDATE` 한 줄로 점유 확인 + 락 획득).
2. "고경합 단일 좌석" 시나리오에서 낙관적 락의 재시도 폭증을 회피한다.
3. 구현·검증이 단순하다(애플리케이션 재시도 로직 불필요). C-2 확정 트랜잭션과 락 경계가 일관된다.

미채택 트레이드오프는 Phase D에서 처리량 한계를 측정한 뒤, 한계 도달 시 Phase E 대기열(ADR-004)로 보완한다.

> **(인간 확정 대기)** — 추천안에 동의 시 위 상태를 `확정`으로 변경하고 이 표기를 제거한다. 다른 전략을 택할 경우 결정·근거를 갱신한다.

## 트레이드오프

- **비관적 락 채택 시:** 고경합 상황에서 락 대기와 커넥션 점유가 처리량 상한을 만든다. Phase D 부하 테스트에서 이 한계를 수치화하고, 한계에 도달하면 대기열(ADR-004) 도입을 재검토한다.
- **낙관적 락/유니크 제약 미채택:** 향후 경합 패턴이 "여러 좌석에 분산되는 저경합"으로 바뀌면 낙관적 락을, 홀드 이력 추적 요구가 생기면(ADR-001 재검토와 함께) 유니크 제약 방식을 다시 검토한다.
