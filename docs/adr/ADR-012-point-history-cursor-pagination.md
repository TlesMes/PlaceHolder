# ADR-012. 포인트 이력 cursor 페이징 채택 + 인덱스 단일 유지

## 상태

채택

## 컨텍스트

Phase E-3에서 사용자의 자기 데이터 조회 API 3종(예약 내역, 포인트 이력, 정산 내역)을 추가했다. 셋 중 **포인트 이력(`point_transactions`)만 본질적으로 append-only**다. 한 사용자의 거래 행은 충전(CHARGE), 예약 확정 시 차감(DEDUCT), 제공자 정산(SETTLE)로 누적되며 영구 삭제되지 않는다. 보관 의무까지 고려하면 사용자당 행 수는 시간이 지남에 따라 단조 증가한다.

반면 예약 내역과 정산 거래는 본질적인 상한이 존재한다. 예약은 사용자가 이벤트에 참여하는 단위로만 늘고, 정산 거래도 제공자가 좌석을 확정받을 때만 늘며 둘 다 사용자당 연 수십~수백 행 수준에 머문다.

조회 응답을 한 번에 모두 반환하면 다음 문제가 발생한다:

- **서버 메모리 압박**: 결과 객체 전부를 메모리에 적재한 뒤 JSON 직렬화 → 사용자당 누적량이 많아질수록 응답당 메모리가 늘고, 동시 요청 시 OOM 위험이 누적과 트래픽의 곱으로 증가
- **DB→앱 전송 비용**: 인덱스 스캔 자체보다 결과 행을 직렬화·전송·역직렬화하는 비용이 더 크고, JPA 엔티티 매핑·컬렉션 적재까지 더해진다
- **응답 지연(TTFB)**: 첫 바이트가 늦어져 사용자 체감 저하
- **운영 시한폭탄**: 데이터가 적을 땐 멀쩡하다가 누적이 임계를 넘는 순간 갑자기 장애가 난다

따라서 append-only 본질을 가진 포인트 이력만 처음부터 페이징을 적용하고, 페이징 방식과 인덱스 전략을 결정한다.

## 근거 (비교 분석)

### 페이징 방식: offset vs cursor

| 항목 | offset(`LIMIT … OFFSET …`) | cursor(`WHERE created_at < ? ORDER BY created_at DESC LIMIT …`) (채택) |
|---|---|---|
| 깊은 페이지 성능 | OFFSET N은 N행을 읽고 버림 → 페이지 깊이에 비례해 느려짐 | "마지막으로 본 시점 이후"라 인덱스에서 바로 진입, 깊이 무관 일정 성능 |
| 전체 페이지 수 | `count(*)` 부수 쿼리로 가능 | 미제공(다음 cursor 유무로만 끝 판단) |
| UX 적합성 | 게시판/검색결과형 페이지 점프에 적합 | 거래 이력/타임라인형 무한 스크롤·최근순 조회에 적합 |
| 데이터 변경 대응 | 페이지 사이 신규 행 삽입 시 중복·누락 발생 | 시간 기준이라 변경에 자연스럽게 강건 |

거래 이력은 사용자가 임의 페이지로 점프하지 않고 "최근부터 순서대로" 보는 패턴이라 cursor가 본질에 맞다. 토스/카카오페이/은행 거래내역, 트위터/인스타 피드 등 append-only + 시간 역순 도메인이 cursor를 표준으로 채택하는 이유와 같다.

### 기간 필터

cursor 외에 `?from=&to=` 쿼리 파라미터를 함께 받고, 미입력 시 기본값을 **최근 3개월**로 채운다.

- 금융권 거래내역 UI의 표준 패턴(기본 1~3개월 + 사용자가 직접 확장)
- "더보기"로 무한히 거슬러 올라가는 대신 자연스러운 상한 부여
- 응답 크기가 시간 누적이 아니라 기간 누적으로 제한됨

### 인덱스: 단일 `(user_id)` vs 복합 `(user_id, created_at)`

cursor 쿼리는 `WHERE user_id = ? AND created_at < ? ORDER BY created_at DESC`이며, 일반적으로 `(user_id, created_at)` 복합 인덱스가 정석이다(equality → range → sort의 컬럼 순서가 정확히 일치, filesort 회피).

하지만 이 도메인에서는 **단일 `(user_id)` 인덱스를 유지**한다.

**근거: 도메인 특성상 사용자당 거래 누적량이 작다.**

좌석 예약 도메인의 거래 발생 트리거:
- 충전(CHARGE): 사용자가 명시적으로 쿠폰 상환 시 — 빈도 낮음
- 차감(DEDUCT): 예약 확정 시 — 사용자가 실제 좌석을 살 때만
- 정산(SETTLE): 제공자가 좌석을 확정받을 때만

대부분의 예약자는 특정 이벤트 보고 단발성으로 가입하는 패턴이 지배적이고, 헤비유저를 가정해도 사용자당 거래는 연 수십~수백 행 수준에 머문다. 토스페이먼츠처럼 사용자당 일 단위로 거래가 쌓이는 도메인이면 복합 인덱스가 필수지만, 이 도메인은 다르다.

복합 인덱스의 이득/비용을 도메인 데이터량에 대입하면:

| 항목 | 사용자당 수십 행 | 사용자당 수만 행 (다른 도메인) |
|---|---|---|
| 인덱스 활용한 정렬 효과 | filesort라도 수십 행 메모리 정렬은 마이크로초 — 체감 무 | 복합 인덱스로 filesort 회피해야 ms 단위 이득 |
| 인덱스 디스크/메모리 비용 | 단일 8byte vs 복합 16byte → 절대량 미미하지만 비례적으로 +50% | 데이터량 자체가 커서 절대 비용도 큼 |
| 쓰기 오버헤드 | 쓰기 빈도 낮은 테이블이라 무시 가능 | 동일 |

**결론**: 도메인이 요구하는 수준에서 단일 `(user_id)`로 cursor 쿼리가 ms 미만에서 완결된다. 복합 인덱스 추가의 이득이 비용 대비 미미하다.

### "측정 없이 인덱스 안 깐다" 원칙과의 정합

기존 인덱스 정책(ADR 별도 명문화 없으나 운영 원칙)은 측정으로 정당화되지 않은 인덱스를 깔지 않는다는 것이다. 본 결정은 그 원칙을 따른다 — 도메인 분석으로 단일 인덱스가 충분함을 보였고, 측정 데이터가 누적되어 단일 인덱스가 불충분함을 보이면 그때 복합으로 전환한다.

## 결정

**포인트 이력 조회에 cursor 페이징(+ 기간 필터)을 채택하고, 인덱스는 기존 단일 `(user_id)`를 유지한다.**

- 엔드포인트: `GET /api/points/history?from=&to=&cursor=&size=`
- 기본값: from=`now - 3개월`, to=`now`, cursor=`to`(미입력 시 가장 최근부터), size=20 (최대 100, 클램프)
- 응답: `items[]` + `nextCursor`(다음 페이지 진입 키, 없으면 null)
- 인덱스: `idx_pt_user_id` 단일 유지 (추가 없음)

예약 내역(`GET /api/reservations/my`)과 정산 거래(`GET /api/providers/my/settlement`)는 페이징 미적용 — 사용자당 행 수의 본질적 상한이 작아 한 번에 반환해도 응답 크기 폭발 위험이 낮다.

다만 정산은 **사업자 측 누적 위험**이 별도로 존재한다. 인기 사업자(다수 이벤트 운영)의 정산 거래는 고객 거래보다 훨씬 빠르게 누적될 수 있어, 본 결정의 가정("사용자당 행 수가 작다")이 약해진다. 본 ADR은 고객 측 가정에 기반한 결정이며, **사업자 측 부담은 별도 측정 PR**에서 다룬다 (`docs/plans/provider-api-index-benchmark.md` 참조). 거기서 데이터 규모를 실제처럼 키워(인기 사업자 시나리오 기준 수십만 행) 단일 vs 복합 인덱스를 측정하고, 정산 API의 cursor 페이징·기간 필터 도입 여부도 함께 결정한다.

## 트레이드오프 / 후속

- **복합 인덱스 전환 조건**: 사용자당 거래량이 수천 행 이상으로 측정되거나, cursor 쿼리 p95가 임계(예: 50ms)를 넘으면 `(user_id, created_at)` 복합 인덱스 추가. Phase D 부하 측정 확장 시 검증.
- **나중에 페이징 도입의 비용 회피**: cursor 페이징을 지금 도입하지 않았다면, 데이터 누적 후 도입 시 응답 구조 변경(`items[]` + `nextCursor`)이 강제되어 프론트·테스트·문서 동시 변경이 필요했다. 처음부터 cursor를 채택해 그 비용을 차단했다.
- **예약 내역의 페이징 도입 가능성**: 본질적 상한이 있다고 가정했지만 헤비유저(인기 이벤트 다수 예약)가 등장하면 재검토. 다만 그 시점에도 페이징 전에 "기간 필터 우선 도입"이 합리적 선택(거래 이력과 다르게 시간 역순 패턴이 약함).
- **size clamp**: 클라이언트가 `size=10000` 같은 값으로 호출해 응답 크기를 늘리는 것을 방지하기 위해 서버에서 100으로 강제 제한.
