# ADR-013. 대기열 설계 — Redis Sorted Set 기반 트래픽 보호 레이어

## 상태

확정 (구현 보류 — 설계 확정, 구현은 Phase E-1)

## 컨텍스트

### ADR-004 재검토 조건 달성

ADR-004는 "부하 테스트로 인프라 한계 수치가 측정된 이후 대기열을 재검토한다"고 명시했다.  
Phase D-2(PR #7, 2026-06-23 머지)에서 이 조건이 달성됐다.

### D-2 측정 결과 요약

- 단일 박스(앱+MySQL+k6 공유)에서 hold p99 knee point: ~880 req/s
- 풀 5→10에서 처리량 2.2배 상승(연결 부족) → 풀 10→20에서 플래토(CPU 한계)
- 적정 풀≈10(기본값), 풀 튜닝으로는 한계 돌파 불가

### 5xx 미발생의 해석 한계

측정 범위에서 5xx는 발생하지 않았지만, 이를 "서버가 과부하를 버텼다"로 해석하면 안 된다.  
k6가 응답 지연 시 요청 생성을 스스로 밀어냈기 때문에(coordinated omission) 서버가 실제 한계까지
부하를 받지 않았다. 즉 **"안 죽는다"도 "죽는다"도 이 측정으로는 결론 낼 수 없다.**

### 그럼에도 대기열이 필요한 이유

실측은 미완이지만, 다음 메커니즘 체인은 성립한다:

1. **knee point는 실측됐다** — 이 지점을 넘으면 지연이 급증한다는 건 수치로 확인됨
2. **knee 이상 지속과부하 → Hikari 대기 → 30초 타임아웃 → 5xx** — Hikari 동작 원리상 필연적 (미실측)
3. **결론:** knee point가 존재하는 이상, 그 이상의 요청을 fast-fail 없이 그대로 흘려보내는 건
   "지연 폭증 → eventually 5xx"로 이어지는 구조다

대기열의 역할은 크래시 방지가 아니라 **knee 이상의 요청을 빠르게 거절(fast-fail)해 나머지 요청의
지연 SLO를 지키는 것**이다. 이는 ADR-004에서 이미 "트래픽 셰이핑 레이어"로 규정한 역할과 동일하다.

## 설계: 대기열을 어디에 거는가

이 ADR의 가장 중요한 결정은 알고리즘이 아니라 **대기열의 경계를 어디에 그을 것인가**다.

### 진입 단위 = 이벤트 (좌석도 전역도 아님)

| 후보 | 동작 | 평가 |
|------|------|------|
| **전역(플랫폼 전체)** | 사이트 들어오는 모든 사용자가 대기 | ❌ 이벤트 B 보러 온 사람이 이벤트 A가 인기라서 대기 — 사용성 최악 |
| **좌석별** | 좌석마다 대기열 분리 | ❌ "A1 줄 / B5 줄" 따로 서기. A1에 100명 대기인데 빈 B5로 못 감 |
| **이벤트별** ✅ | 이벤트 하나당 대기열 하나, 그 안 좌석은 공유 | 입장하면 빈 좌석 아무거나 선택. 인기 좌석 경합은 락이 처리 |

→ **`queue:{eventId}` 하나를 그 이벤트의 모든 좌석이 공유한다.** 대기열은 "이벤트 N의 hold 영역에 들어갈 권리"를 통제하지, 특정 좌석을 통제하지 않는다.

### 2층 구조: 대기열(트래픽) vs 락(정합성)

| 레이어 | 단위 | 막는 것 | 담당 |
|--------|------|---------|------|
| **대기열** | 이벤트 | 동시에 몇 명이 이 이벤트에서 hold를 시도하나 | 이 ADR-013 |
| **비관적 락** | 좌석 | 그 안에서 좌석 A1을 정확히 한 명만 잡나 | ADR-008 |

입장 토큰을 받은 8명이 동시에 같은 좌석 A1을 노려도, 그건 대기열이 아니라 **락**이 정리한다(한 명 성공, 나머지 "이미 점유됨"). 대기열은 정합성이 아니라 **트래픽 셰이핑**만 책임진다.

### 거는 위치 = hold 진입점 (조회 페이지 아님)

- **좌석 그리드 조회**(`GET /api/events/{id}/seats`)는 읽기 → **대기열 없음.** 막으면 "원하는 좌석이 비었는지"조차 못 보고 대기하는 꼴. 실제 티켓팅도 배치도는 먼저 보여준다.
- **`POST /api/seats/{seatId}/hold`** — 좌석을 실제로 잡는 행동, 비관적 락·커넥션 점유·D-2 부하가 발생하는 그 지점. **여기가 대기열 게이트.**
- **confirm은 별도 게이트 불필요** — confirm은 hold 성공자만 호출하므로(confirm 시도자 ⊆ hold 통과자) hold를 막으면 자연히 제한된다.

```
좌석 그리드 구경 (GET seats)          ← 자유, useSeatPolling 실시간 갱신
   │  "예매하기" 클릭
   ▼
[대기열 게이트] queue:{eventId}         ← ✅ 여기
   │  입장 토큰 있으면 통과
   ▼
좌석 잡기 (POST hold)                  ← 비관적 락 발생
```

### 선택적 활성화 (인기 이벤트만)

모든 이벤트에 대기열을 거는 건 오버엔지니어링이다(좌석 5개·예약자 3명 이벤트엔 불필요). 이벤트에 `queue_enabled` 플래그를 두거나, 동시 hold 시도가 임계를 넘으면 자동 활성화한다. 이것이 "페이지는 자유롭게 접근되지만 인기 이벤트의 예약만 대기열 경쟁"을 가능하게 하는 장치다.

### 전체 흐름

```
[사용자] ──hold 요청──▶ [대기열 API]
                              │
                    대기열 진입 (ZADD)
                    순번 반환
                              │
              ◀── 순번/상태 폴링 ──
                              │
                    입장 차례 도달
                    입장 토큰 발급
                              │
         입장 토큰 + hold 요청
                              │
                    토큰 검증 통과
                              ▼
                       [Hold API] ──▶ 기존 비관적 락 흐름
```

### Redis Sorted Set 구조

```
키:    queue:{eventId}
score: 진입 시각 (Unix timestamp ms) — FIFO 보장
member: userId
```

| 연산 | Redis 명령 | 용도 |
|------|-----------|------|
| 진입 | `ZADD queue:{eventId} NX <ts> <userId>` | 중복 진입 방지(NX) |
| 순번 조회 | `ZRANK queue:{eventId} <userId>` | 현재 대기 순번 |
| 입장 처리 | `ZPOPMIN queue:{eventId} <N>` | N명씩 배치 입장 |
| 대기열 크기 | `ZCARD queue:{eventId}` | 전체 대기 인원 |

### 입장 토큰

- 입장 허가된 사용자에게 **단기 토큰**(Redis key: `entry:{eventId}:{userId}`, TTL=5분) 발급
- Hold API 진입 시 이 토큰 존재 여부 확인 → 없으면 대기열로 리다이렉트
- 토큰 TTL = 홀드 TTL(5분)과 동일 — 토큰 만료 후 홀드를 시도하면 대기열 재진입

### 배치 입장 스케줄러 — Lua 원자 입장 (ceiling + rate)

입장 제어는 두 레버를 함께 쓴다(현업 대기실 표준).
- **ceiling (C):** 동시 활성 세션(입장 토큰 보유) **전역** 상한. 막는 대상은 *이벤트*가 아니라 *인스턴스 용량*이므로 이벤트 수와 무관한 전역 값이다(앱 세션 용량 기준 — DB 풀 크기 아님).
- **rate (R):** 초당 입장 허용 수. 빈자리가 한꺼번에 생겨도 유입을 평탄화한다.

핵심 연산(활성 수 확인 → ZPOPMIN → 토큰 발급)은 **Lua 스크립트(`admit.lua`)로 원자화**한다.

```
admit(eventId, now, ceiling, ratePerSec, ttlMs, max):   // EVAL 1회 = 원자
  ZREMRANGEBYSCORE active:all 0 now          // 만료 활성 청소
  while admitted < max:
    if ZCARD active:all >= ceiling: break              // C
    if GET rate:{now/1000} >= ratePerSec: break        // R
    user = ZPOPMIN queue:{eventId} 1; if none: break
    INCR rate:{now/1000}; EXPIRE rate:{...} 2
    ZADD active:all (now+ttlMs) "{eventId}:{user}"      // 활성 등록
    SET entry:{eventId}:{user} 1 PX ttlMs               // 게이트용 토큰
  return admitted

매 1초(스케줄러): for each active eventId: admit(eventId, ..., max=ratePerSec)
enter() fast-path:                              admit(eventId, ..., max=1)
```

**왜 Lua(원자 연산) — 락 아님:** check-then-act을 EVAL 1회로 묶으면 Redis의 단일 스레드 직렬 실행이 동시 호출을 자동 직렬화한다. 그래서 **다중 인스턴스/스레드(scale-out, fast-path)에서도 ceiling·rate 초과가 불가능**하다. SETNX 분산 락은 같은 효과를 내지만 락 lifetime(크래시 시 미해제, TTL 만료 중 이중 보유) 함정이 있어, 임계구역을 한 원자 연산으로 만드는 Lua를 택한다. 단일 스레드 스케줄러만 있던 이전 설계는 단일 인스턴스에서만 캡이 지켜졌다(다중 인스턴스 시 over-admit).

**왜 ceiling이 전역인가:** 이전 `max_concurrent_holds`는 입장 토큰(TTL 5분) 수를 *이벤트별*로 세며 값을 DB 풀(8)에 맞췄다 → 사실상 "5분에 8명"이라는 비현실적 입장 속도 + 이벤트 N개면 8N으로 인스턴스 용량 초과. hold는 짧은 트랜잭션(C-1: 락→상태변경→즉시 커밋)이라 풀이 막을 건 "동시 hold 트랜잭션"이지 "5분 토큰"이 아니다. 그래서 ceiling은 앱 세션 용량 기준의 전역 값으로 두고, rate를 별도 레버로 분리한다. 초기값은 운영 측정으로 조정(`max-active-sessions`, `rate-per-second`).

### 대기 상태 조회 API

- `POST /api/queue/{eventId}/enter` — 대기열 진입, 현재 순번 반환
- `GET  /api/queue/{eventId}/status` — 현재 순번 + 입장 토큰 존재 여부 조회
- 프론트엔드는 기존 `useSeatPolling` 패턴과 동일하게 폴링으로 구현 (추후 SSE 교체 가능)

## 채택 근거: Redis Sorted Set vs 대안

| 후보 | 장점 | 단점 |
|------|------|------|
| **Redis Sorted Set** | ZADD/ZRANK/ZPOPMIN O(log N), TTL 네이티브, 서버 재시작 후 대기열 유지 | Redis 인프라 추가 |
| RDB(MySQL) | 별도 인프라 없음 | 순번 조회 폴링 부하, 실시간 통지 어색, 인덱스/락 복잡 |
| JVM 인메모리 | 빠름, 설정 없음 | 서버 재시작 시 대기열 유실, 멀티 인스턴스 불가 |

ADR-004에서 이미 Redis Sorted Set을 우선 검토 대상으로 명시했고, 위 비교로 동일 결론에 도달한다.

## 결정

**Redis Sorted Set 기반 대기열을 채택한다.**

- 역할: 트래픽 보호(fast-fail) — 동시성 정합성은 여전히 비관적 락(ADR-008)이 보장
- 진입 단위: `eventId`별 독립 대기열 (좌석은 그 대기열을 공유)
- 거는 위치: hold 진입점(`POST /seats/{id}/hold`)만. 조회는 자유, confirm은 hold 하위집합이라 불필요
- 활성화: 인기 이벤트만 선택적(`queue_enabled`)
- 입장 단위: 배치(슬롯 여유분만큼), TTL=5분 입장 토큰으로 Hold API 접근 통제
- 대기 상태 통지: 폴링 → 추후 SSE 교체 가능(ADR-004 언급 방향과 동일)

## 트레이드오프와 열린 문제

**채택 시:**
- Redis 인프라가 추가된다. 단일 박스 개발 환경에서는 Docker로 띄운다.
- 배치 입장 스케줄러가 새 장애 포인트가 된다 (Redis 다운 → 대기열 불가).
  → fallback: Redis 다운 시 대기열 없이 직접 hold 허용 (기존 동작으로 강등).

**⚠️ 알려진 한계 — per-event 슬롯 합이 전역 풀을 초과할 수 있음:**

대기열은 `eventId`별로 슬롯(`max_concurrent_holds`)을 관리하지만, 실제로 보호하려는 자원(D-2 knee ~880/s, 커넥션 풀 10, 박스 CPU)은 **이벤트와 무관한 전역 예산**이다. 따라서 이벤트가 동시에 여러 개 달아오르면 각 이벤트가 슬롯을 지켜도 `5개 × 8명 = 40`이 전역 풀 10에 몰려 박스가 다시 한계를 넘을 수 있다.

이 ADR은 **per-event 대기열만 채택하고 이 모순을 의도적으로 남긴다.** 근거:
- 현실에서 메가 이벤트(콘서트 티켓 오픈 등)는 한 타임에 동시 다발하지 않고 하나가 트래픽을 독점한다. 그 순간의 핫 이벤트 1개 ≈ 전역 트래픽이 되어, per-event 대기열이 사실상 전역 throttle로 동작한다.
- 단일 박스 포트폴리오 범위에서 "동시 핫이벤트 다수" 시나리오는 과설계다.
- 진짜 필요해지면 **전역 입장 예산**(전역 동시 hold 수를 두고 per-event 큐가 그걸 나눠 씀)으로 보완 — 향후 과제.

**열린 문제 (구현 시 결정):**
- `max_concurrent_holds` 초기값 튜닝 — D-2 knee point를 별도 머신 환경에서 재측정하면 더 정확해짐
- 이벤트 종료 후 대기열 TTL 정리 전략
- Redis 연결 실패 시 graceful degradation 범위
