# PlaceHolder — 마스터 플랜 (Claude Code 작업 기준)

> 기간: 2026.05 ~ 2026.10 (5개월), 백엔드 단독
> 핵심 주제: 동시성 제어 + 개발 프로세스 성숙도 + 도메인의 법적 맥락 이해
> 포트폴리오 타겟: 실시간 처리/동시성이 핵심인 서비스 회사

---

## 작업 원칙

1. **모든 단계는 PR 단위로 진행** — main 직접 push 금지. feature 브랜치 → PR → self review → merge
2. **결정 지점마다 best practice 탐색** — Claude Code와 트레이드오프 논의 후 ADR로 기록
3. **포트폴리오 요소는 의도적으로 선택** — "domain → problem → solution → result" 구조로 정리 가능한 작업 우선
4. **핵심 로직은 반드시 테스트** — 특히 동시성/트랜잭션
5. **AI 활용 경계 준수** — 설계 판단/개념 이해는 직접, 보일러플레이트는 AI

## Phase B — 기반 인프라 (6월 초)

### B-1. Repository 계층
- 7개 엔티티에 대한 JpaRepository 인터페이스
- 도메인별 패키지 구조 유지 (domain/{name}/repository/)

### B-2. 인증/인가 (JWT)
- Phase 1에서 학습한 JWT 이식 (jjwt 0.12.3)
- 회원가입 (역할 선택: BOOKER/PROVIDER)
- 로그인 → JWT 발급
- JwtAuthenticationFilter + SecurityConfig
- 역할 기반 접근 제어 (@PreAuthorize)
- **포트폴리오 포인트:** Spring Security Filter Chain 구조 이해

### B-3. 회원가입 시 계정 생성 트랜잭션
- BOOKER 가입 → User + BookerAccount 동시 생성
- PROVIDER 가입 → User + ProviderAccount 동시 생성
- **결정 지점:** 계정 생성을 회원가입 트랜잭션에 포함할지, 이벤트로 분리할지 (ADR 후보)

### B-4. 기본 CRUD
- 제공자: 이벤트 등록 (좌석 메타데이터 일괄 생성)
- 예약자: 이벤트 목록 / 좌석 목록 조회
- **결정 지점:** 좌석 일괄 생성 방식 (반복 INSERT vs batch insert) — 성능 측정 후 판단

## Phase C — 핵심 루프 (동시성 제어) ★ 프로젝트의 심장

### C-1. 홀드 (좌석 점유)
- 같은 (event, seat)에 동시 요청 시 한 명만 성공
- **결정 지점 (핵심 ADR):** 락 전략 선택
  - 비관적 락 (SELECT ... FOR UPDATE)
  - 낙관적 락 (@Version)
  - DB 유니크 제약 활용
  - → 각 방식 구현 후 트레이드오프 비교, ADR로 근거 문서화
- 홀드 시 status=HELD, held_by, held_until(TTL) 설정
- **포트폴리오 포인트:** "어디에, 어떤 락을, 왜" 명확히 설명 가능

### C-2. 확정 (트랜잭션 원자성)
- [잔액 검증 → 예약자 포인트 차감 + 제공자 정산예정액 적립 + 좌석 CONFIRMED + Reservation 생성 + PointTransaction 2건]
- 단일 @Transactional
- 잔액 부족 시 홀드 해제 후 실패 반환
- **결정 지점:** 트랜잭션 전파(propagation)/격리(isolation) 레벨 선택 (ADR 후보)
- **포트폴리오 포인트:** 두 계정에 걸친 정합성, @Transactional 실전

### C-3. 자동 만료 해제
- held_until 경과 시 status=AVAILABLE 복귀
- **결정 지점:** 만료 처리 방식
  - 스케줄러 폴링 (주기적 스캔)
  - 조회 시점 lazy 만료 처리
  - → 트레이드오프 비교 (ADR 후보)
- idx_seats_status_held_until 인덱스 활용

### C-4. 동시성 정합성 테스트 ★ 핵심 어필
- 동시 N건 요청 시 좌석/포인트/정산이 깨지지 않음 증명
- CountDownLatch / ExecutorService 활용한 동시 요청 시뮬레이션
- **포트폴리오 포인트:** "동시성을 테스트로 증명했다"

---

## Phase D — 부하 테스트 + 측정 ★ 측정→판단 서사

### D-1. 부하 테스트 환경 구축
- 도구 선택 — **결정 지점**
- 동시 점유 시나리오 스크립트 작성

### D-2. 측정
- 95th %ile 응답시간, RPS, 에러율
- 동시 접속 수 N명 기준 hold/confirm 경합 시나리오
- **포트폴리오 포인트:** "동시성 정합성을 테스트로 증명하고, 수치로 한계를 측정했다"

### D-3. 한계 분석
- RPS 임계점 측정 → 병목 지점 식별
- **포트폴리오 포인트:** "측정으로 한계를 알고, 다음 결정의 근거로 삼았다"
- → 이 수치가 Phase E 대기열 도입 판단의 근거

---

## Phase E — 조건부 확장 (측정 기반)

> Phase D 수치 없이는 착수 금지. 우선순위 판단 후 선택적 진행.

### E-1. 대기열 (트래픽 보호 레이어) — 1순위 후보
- 전제: D-3 한계 수치로 필요성 입증
- Redis Sorted Set 순번 + 입장 토큰 + 대기 상태 통지
- **ADR:** "대기열 ≠ 경합 해결, = 트래픽 보호" 구분
- **포트폴리오 포인트:** 측정→판단→설계 완결 서사. 구현 못해도 설계안까지면 OK

### E-2. 테스트 PG 연동 (토스페이먼츠 샌드박스) — 2순위 후보
- 예약자 포인트 충전 (현금→포인트)
- 결제창 → 승인 → 웹훅 플로우
- 멱등성 키, 웹훅 기반 확인
- **ADR:** 동기 승인 vs 웹훅 비동기
- **포트폴리오 포인트:** 결제-포인트 정합성, RestClient 직접 연동(샘플 복붙 아님)

### E-3. 조회 기능 (완성도용) — 3순위
- 예약 내역 / 포인트 내역 / 정산 내역
- N+1 의식한 fetch join 적용
- **포트폴리오 포인트:** N+1 해결 (fetch join 적용 전후 쿼리 수 비교)

## Phase F — 배포 + 문서 마감

### F-1. CI/CD
- GitHub Actions: 빌드 + 테스트 자동 실행 (CI)
- AWS EC2 배포 (CD) — TeamMoa 경험 활용, 시간 여유 시
- Dockerfile + docker-compose app 서비스 추가

### F-2. 프론트엔드 (백엔드 검증용)
- 좌석 그리드 UI (AVAILABLE/HELD/CONFIRMED 색상 구분)
- 코드 에이전트 활용, 백엔드 검증 범위 내로 제한

### F-3. 포트폴리오 문서 마감
- README 완성 (실행 방법, 아키텍처 다이어그램)
- 각 핵심 작업을 domain→problem→solution→result로 정리
- ADR 전체 정리
- 면접 스토리 2종 다듬기 (동시성 설계 결정 서사, 정산 법적 맥락)

- [ ] 결제-포인트 정합성/멱등성 (E-2)
- [ ] N+1 해결 (E-3)
- [ ] CI/CD 파이프라인 (F-1)
- [ ] 정산의 법적 맥락 이해 (전 과정, README)

## 진행 방식

1. 각 단계 시작 전 이 프로젝트(포트폴리오 관리)에서 결정 지점 논의
2. 결정 후 Claude Code에서 구현
3. PR 생성 → self review
4. 결정 근거는 ADR로 기록
5. 완료 시 Notion 일지 기록 (다음 할 것 명시)
