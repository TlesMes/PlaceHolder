# Phase D-2 — hold 처리량 한계 측정 (종료)

> 작성: 2026-06-10 / 갱신: 2026-06-23 / 브랜치: `feature/loadtest-hold-confirm-knee` (Draft PR #7)
> **D-2 측정 종료(2026-06-22).** confirm은 미실행(사유는 「정직한 한계」).

## 측정 환경
- CPU: AMD Ryzen 5 5600 (6코어/12스레드) / RAM 32GB / MySQL 8.0.46 (Docker, `placeholder-mysql`:3306) / k6 v2.0.0 (Windows)
- **앱 + MySQL + k6가 한 PC에서 같이 실행됨** (그래서 절대 수치 비교는 무의미 — 곡선 모양·상대 비교만 본다)
- 앱 기동: `./mvnw.cmd spring-boot:run "-Dspring-boot.run.profiles=local,loadtest"` (`local`=DB자격증명, `loadtest`=쿼리통계+쿠폰컨트롤러+Hikari DEBUG)
- 커넥션 풀 크기: `HIKARI_POOL` 환경변수로 바꿀 수 있음(기본 10). 각 측정 후 앱 재시작(ddl-auto: create로 DB 초기화 → k6 setup이 매번 데이터 새로 만듦)

## 측정 기준 = 성공 요청의 지연 p99
초당 요청 수(도착률)를 점점 올리며, **성공한 hold 응답의 p99 지연이 임계(기본 1초)를 넘는 순간**을 한계점으로 보고 자동 중단한다.
- 이 시스템은 과부하 때 에러(5xx)로 죽지 않고 "지연이 늘거나 4xx 거절"로 버텨서, **5xx 기준으로는 한계를 못 잡음**(1차에서 확인) → 지연 기준으로 바꾼 것.
- `hold_duration`은 성공(200)일 때만 집계 → 빠른 4xx 거절이 지연에 안 섞임. 5xx 발생 시 즉시 중단은 보조로 유지.

## 측정 여정
| 회차 | 설정 | 결과 | 판정 |
|---|---|---|---|
| 1차 | 도착률 50→5000, 좌석 2만 | 거절 75%, 5xx 0, p95 6.35s, k6가 못 보내고 버린 요청 235,781건, 중단 안 걸림 | 상한 너무 높음 + 좌석 부족 → 무효 |
| 2차 | 도착률 50→600, 좌석 2만 | 성공 19,262 ≈ 좌석 수 → **좌석이 다 떨어져 측정 오염**(거절 70%) | 무효 |
| 3차 | 도착률 200→1500, 좌석 7.5만 | 좌석 부족 해소, 한계점에서 지연 기준 정상 중단 | 유효 |
| 풀 크기별 | 풀 5/10/20 (+CPU 측정) | 병목 식별 (아래) | — |

## 풀 크기별 측정 (한계점 = 성공 p99가 1초 넘는 도착률)
| 풀 | 한계 도착률 | p99 | 그때 CPU | 커넥션 상태 |
|----|------|-----|------|--------|
| 5  | ~390/s | 1.17s | **~65% (놀고 있음)** | 5개 다 사용중, 대기 195 |
| 10 | ~840/s | 1.04s | 미측정(~90% 추정) | 10개 다 사용중, 대기 190 |
| 20 | ~890/s | 1.16s | 미측정(~100% 추정) | 20개 다 사용중, 대기 181 |

- **풀 5→10: 한계가 2.2배로 늘어남(연결이 부족했다는 뜻) / 10→20: 1.06배(거의 안 늘어남).**
- 풀 5는 CPU가 65%로 놀고 있는데도 꺾임 = **연결이 부족해서** 못 늘어난 것. 풀 10부터는 **CPU**가 한계.

## 결론
1. **적정 풀 ≈ 10**(HikariCP 기본값). 그보다 적으면 연결 부족, 많으면 의미 없음(지연만 나빠짐). → **튜닝할 필요 없고 기본값이 맞았다.**
2. **시험한 부하 범위에선 5xx가 안 났다** — 풀(10)이 꽉 차면 초과 요청이 **큐에서 대기**하며 에러 대신 지연으로 흡수했기 때문. **단 "5xx로 안 죽는다"는 보장이 아님:** 과부하를 더 오래/세게 지속하거나 실제 사용자처럼 무제한으로 들이닥치면, 커넥션 대기가 HikariCP 30초 제한을 넘겨 **5xx가 날 것**(이어서 Tomcat 처리 한계도 초과). 한계는 **이 PC의 CPU(~880/s)**지 동시성 로직이 아님.
3. **대기열(E-1) 의미:** 지금은 풀이 *그냥* 큐로 받아내는 거라 지연이 끝없이 늘 수 있음 → E-1은 이걸 **"초과 요청을 빨리 거절(fast-fail)해서 지연을 일정 이하로 지키는" 능동 제어**로 바꾸는 것. ADR "대기열 = 트래픽 보호"와 일치.

## 정직한 한계 (재현·해석 시 필독)
- **부하를 좌석 전체에 골고루 분산시켜 측정** → 잰 것은 "앱+DB의 처리량"이지 "같은 좌석에 몰릴 때 락이 버티는가"가 **아니다.** 경합(같은 좌석 경쟁)의 정확성은 JUnit C-4가 이미 증명. → 락 설계 자체의 부하 거동은 **소수 좌석에 다수가 몰리는 시나리오**가 별도로 필요(미수행, 백로그).
- **5xx가 안 난 데엔 측정 한계도 있음**: ① 지연 한계점에서 자동 중단해 과부하를 오래 못 끌었고, ② **k6가 못 보내고 버린 요청(dropped_iterations) 때문에 서버가 과부하를 다 안 받음.** 실제 사용자는 그런 게 없어 계속 들이닥침 → 결론 2의 5xx 시나리오로 감.
- 절대 수치(~880/s)는 한 PC에서 k6까지 같이 돌린 값이라 **다른 환경엔 못 옮김 — 곡선 모양만 의미 있음.**
- CPU는 풀 5만 측정(10/20은 추정).
- **confirm.js 미실행:** hold가 "한계는 인프라/CPU지 로직 아님"으로 결론나, confirm도 같은 양상일 것으로 보여 추가 측정 가치가 낮다고 판단해 보류.

## D-2 종합 (D-1 대비)
- **성능 면의 실질 성과는 D-1(N+1 해결)** — 쿼리 1+2N→2로 줄이고 개선을 수치로 증명.
- **D-2는 보조 결론** — "한계는 인프라/CPU지 동시성 로직이 아니다 + 풀이 큐로 버틴다." 측정이 가설을 3번 교정(풀이 병목→아니다→10 미만에선 맞다)한 과정 자체가 산출물.
