# ADR-002. PointAccount — 역할별 테이블 분리

## 상태
확정

## 컨텍스트
예약자의 포인트 잔액(`balance`)과 제공자의 정산 예정액(`settlement_balance`)을 하나의 테이블에 통합하거나, 역할별로 분리하는 두 가지 방식이 있다.

## 결정
`booker_accounts`와 `provider_accounts`를 별도 테이블로 분리한다.

## 근거
- 두 값의 성격이 다르다. `balance`는 예약자가 소비하는 포인트이고, `settlement_balance`는 제공자가 정산받는 예정액 집계다. 현금화 가능 여부도 다르다.
- 이 서비스는 역할이 명확히 분리된 구조다. 역할 전환이 자유로운 당근마켓형 구조가 아니다.
- 정산 트랜잭션 코드에서 `providerAccount.addSettlement(amount)` 처럼 의도가 명확하게 드러난다.
- 통합 방식은 예약자가 `settlement_balance` 컬럼을, 제공자가 `balance` 컬럼을 항상 0으로 갖게 되어 도메인적으로 어색하다.

## 트레이드오프
- 테이블이 하나 더 늘어난다.
- 한 사람이 예약자+제공자 역할을 동시에 갖는 시나리오에서는 두 계정을 모두 생성해야 한다. 단, 이 프로젝트는 Enum 단일 역할(ADR-003)로 설계되어 해당 시나리오가 발생하지 않는다.
