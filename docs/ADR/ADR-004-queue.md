# ADR-004. 대기열 — ERD 설계 단계에서 제외

## 상태

확정 (재검토 예정)

## 컨텍스트

티켓팅 시스템에서 트래픽 폭주 시 인프라를 보호하기 위한 대기열(Queue) 도입을 검토했다.
저장소 선택지로 RDB(MySQL)와 Redis Sorted Set이 있었다.

## 결정

현 단계에서 대기열을 ERD에 포함하지 않는다.

## 근거

- 대기열은 동시성 제어(락 경합 해결)가 아니라 트래픽 보호(트래픽 셰이핑) 레이어다. 역할이 다르다.
- 핵심 루프(홀드 → 확정 → 만료 해제) 완성 전에 대기열을 설계하면 필요성이 검증되지 않은 기능에 설계 복잡도가 추가된다.
- 도입 순서: 핵심 루프 구현 → 부하 테스트(RPS 임계점 측정) → 수치 기반 필요성 판단 → 설계/구현.

## 재검토 조건

부하 테스트에서 인프라 한계 수치(RPS 임계점)가 측정된 이후. 도입 시 Redis Sorted Set 방식을 우선 검토한다.

- Redis Sorted Set: TTL 네이티브, 순번 조작 O(log N), 실시간 대기 순번 통지(SSE/WebSocket) 연동에 적합.
- RDB 방식은 폴링 기반이 되어 실시간 대기 순번 통지가 어색하다.
